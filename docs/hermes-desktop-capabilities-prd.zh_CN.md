# Hermes Desktop 风格能力中心 PRD（中文版）

本文定义白泽 AI 平台 Web 版 HermesAgent 的下一阶段产品需求：在现有 HermesAgent 聊天体验基础上，补齐接近 Hermes Desktop 的能力管理入口，重点覆盖技能、工具和消息平台。首期目标是让用户能在浏览器中创建并查看自己的 Hermes 技能，了解当前可用工具，并通过微信扫码把 HermesAgent 接入个人微信会话。

## Problem Statement

当前 Web 版 HermesAgent 已具备基础聊天、会话列表、会话操作、`/new`、`/save`、`/retry` 和 `/skill` 创建能力，但用户仍然缺少一个稳定的能力管理界面。典型问题包括：

- 用户创建技能后，只能通过提示词或后续对话间接确认，无法像 Hermes Desktop 一样在界面中看到“我创建了哪些技能”。
- 工具能力不透明，用户不知道当前 HermesAgent 能访问哪些工具、哪些工具集已启用、哪些工具需要额外配置。
- 平台管理员和用户无法在 Web 界面中管理消息平台接入，尤其是无法通过扫码流程把 HermesAgent 连接到微信。
- 多用户 sidecar 场景下，技能、工具和消息平台必须按平台用户隔离，不能因为共用一个 Hermes 容器而互相看见或复用敏感状态。
- 微信扫码流程涉及二维码、轮询、会话状态、登录态失效和重连，如果没有产品化入口，用户只能依赖运维手工配置，交付成本高。

## Solution

在 HermesAgent 下新增“能力中心”体验，包含三个首期模块：

1. 技能：支持创建 Hermes skill，显示用户自己创建的技能，并可查看内置技能与个人技能的来源差异。
2. 工具：展示当前 Hermes API Server 平台可用的工具集、工具列表、启用状态和配置状态，让用户知道 HermesAgent 实际可调用什么。
3. 消息平台：首期支持微信扫码接入。用户在 Web 页面发起连接，平台展示二维码并轮询状态；扫码成功后，Hermes sidecar 按该平台用户保存微信会话凭据，并把后续微信消息路由到该用户的 HermesAgent 会话。

该能力中心不替代平台的模型权限、团队额度、API 密钥、日志和审计。用户访问 HermesAgent 仍然经过白泽 AI 平台登录态、模型授权和平台治理；Hermes API Server 密钥继续只存在于服务端环境变量或容器内部，不下发到浏览器。

## User Stories

1. 作为普通用户，我希望在 HermesAgent 页面看到“技能”入口，以便管理可复用的任务能力。
2. 作为普通用户，我希望通过表单创建 Hermes skill，以便把常用工作流程沉淀成可复用说明。
3. 作为普通用户，我希望创建技能时填写名称、分类、描述和执行说明，以便 Hermes 能在合适场景自动理解技能用途。
4. 作为普通用户，我希望技能创建后立即出现在“我的技能”列表，以便确认保存成功。
5. 作为普通用户，我希望列表区分“我的技能”和“内置技能”，以便知道哪些技能属于自己、哪些来自系统。
6. 作为普通用户，我希望只看到自己创建的技能，以便多人共用 sidecar 时不会暴露其他用户的工作流程。
7. 作为普通用户，我希望能按名称、分类和描述搜索技能，以便快速找到目标技能。
8. 作为普通用户，我希望能查看某个技能的详情，以便确认它的触发场景和执行步骤。
9. 作为普通用户，我希望能复制技能名称或引用方式，以便在聊天中明确要求 Hermes 使用某个技能。
10. 作为普通用户，我希望看到技能保存失败的具体原因，以便修正 YAML frontmatter、名称格式或必填字段。
11. 作为普通用户，我希望技能列表能在创建后自动刷新，以便不需要手动重载页面。
12. 作为普通用户，我希望能看到当前 HermesAgent 可用的工具集，以便了解它能否访问浏览器、文件、命令、计划任务或消息平台能力。
13. 作为普通用户，我希望工具集显示启用状态，以便知道哪些工具当前真的会被 Hermes 使用。
14. 作为普通用户，我希望工具集显示配置状态，以便知道某个工具不可用是因为未启用还是缺少密钥/配置。
15. 作为普通用户，我希望展开工具集后看到具体工具名，以便理解 HermesAgent 可以调用的操作边界。
16. 作为普通用户，我希望工具列表保持只读起步，以便先降低误配置风险。
17. 作为平台管理员，我希望后续可以开放工具集启停和配置入口，以便按团队或部署环境控制能力范围。
18. 作为普通用户，我希望在 HermesAgent 中看到“消息平台”入口，以便把 HermesAgent 接入外部聊天软件。
19. 作为普通用户，我希望首期支持微信扫码，以便像使用桌面端一样把个人微信接入 HermesAgent。
20. 作为普通用户，我希望点击“连接微信”后看到二维码，以便使用微信扫码登录。
21. 作为普通用户，我希望二维码页面显示扫码中、已扫码待确认、已连接、已过期和失败状态，以便知道下一步该做什么。
22. 作为普通用户，我希望二维码过期后能重新生成，以便无需刷新页面重新开始。
23. 作为普通用户，我希望微信连接成功后看到当前连接账号的脱敏标识，以便确认连接的是正确账号。
24. 作为普通用户，我希望能断开微信连接，以便在不需要时撤销消息平台接入。
25. 作为普通用户，我希望微信会话与我的平台账号绑定，以便其他平台用户不能复用我的微信登录态。
26. 作为普通用户，我希望微信消息进入 HermesAgent 后沿用我的模型权限和额度策略，以便使用行为仍由平台治理。
27. 作为平台管理员，我希望可以关闭微信扫码入口，以便未启用 Hermes weixin 依赖或不允许个人微信接入的部署不会暴露入口。
28. 作为平台管理员，我希望看到微信接入的运行状态和错误信息，以便定位依赖缺失、网络失败、扫码超时或凭据失效问题。
29. 作为平台管理员，我希望微信凭据只保存在 Hermes sidecar 的用户隔离目录中，以便避免浏览器或平台数据库保存敏感会话信息。
30. 作为平台管理员，我希望平台记录消息平台连接、断开和错误事件，以便满足审计要求。
31. 作为平台管理员，我希望 API Server 对技能、工具和消息平台接口统一使用服务端 Hermes 密钥转发，以便普通用户无法绕过平台直接访问 Hermes。
32. 作为平台管理员，我希望所有 Hermes 能力接口都写入平台用户 ID 和会话 ID，以便 sidecar 内部可以按用户隔离技能目录、会话和消息平台状态。
33. 作为交付工程师，我希望部署文档说明需要的 Hermes 镜像版本和环境变量，以便 partner 私有化部署时不会遗漏微信扫码或技能接口。
34. 作为交付工程师，我希望功能可在无微信配置时降级为不可用状态，以便不影响 HermesAgent 聊天主流程。

## Implementation Decisions

- Web 版 HermesAgent 保留聊天主界面，在同一一级入口下增加能力中心区域。能力中心可以是页内侧栏、顶部标签或独立子路由，但应保持与现有 HermesAgent 会话树、命令输入和模型选择体验一致。
- 技能模块首期使用现有 Hermes API Server 能力：创建走平台后端代理到 `POST /v1/skills`，列表走平台后端代理到 `GET /v1/skills`。
- 技能列表必须支持来源字段。至少区分用户自建技能、系统内置技能和未来团队共享技能。若 Hermes 当前返回值不足，需要在 Hermes API Server 或平台代理层补充可稳定消费的 `source`、`owner_scope`、`category`、`path`、`description` 等字段。
- “我的技能”只展示当前平台用户隔离目录下的技能。隔离键使用服务端登录态生成的 Hermes 用户 ID，不能由前端传入任意用户 ID。
- 技能创建表单继续由前端生成合法 `SKILL.md` 内容，内容必须包含 YAML frontmatter。后端应透传 Hermes 的校验错误，并转成用户可理解的错误提示。
- 工具模块首期以只读展示为主。平台代理调用 Hermes `GET /v1/toolsets`，展示工具集名称、标签、描述、是否启用、是否配置完成，以及展开后的工具名列表。
- 工具模块不在首期直接开放任意工具安装、任意命令配置或 MCP 动态写入。后续如需开放，应增加管理员审批、配置校验、审计和回滚能力。
- 消息平台模块首期只做微信扫码。产品文案使用“微信扫码接入 HermesAgent”，避免与平台已有“微信登录”“微信支付”混淆。
- 微信扫码能力基于 Hermes 的 `weixin` 平台适配器。平台后端需要为 Web 提供受控代理接口，用于创建二维码、查询二维码状态、查看连接状态和断开连接。
- 微信扫码接口不应直接暴露 iLink、Hermes 内部密钥或微信会话 token。浏览器只接收二维码图片/二维码 URL、状态枚举和脱敏账号信息。
- 微信连接状态需要按平台用户隔离存储。推荐落到 Hermes sidecar 的用户目录，例如基于 `X-Hermes-User-Id` 的 Hermes home 子目录，而不是全局默认 profile。
- 微信消息进入 Hermes 后，应沿用该用户的 Hermes 会话隔离策略。用户从微信发出的消息应能被映射到该用户的 HermesAgent 身份，而不是进入共享默认会话。
- 平台后端需要为 Hermes 能力代理设置统一超时、错误映射和审计事件。技能和工具属于短请求；微信扫码状态轮询应限制频率并支持二维码过期。
- 如果 Hermes sidecar 未启用、`HERMES_API_SERVER_KEY` 缺失、`HERMES_API_SERVER_URL` 不可达或 Hermes 镜像版本不支持对应接口，能力中心应显示明确的不可用状态，不影响聊天页面加载。
- 部署配置继续使用 `version.ini` 中的 `image_name_hermes` 和 `hermes_versions` 管理 sidecar 镜像。微信扫码能力需要在 Hermes 镜像中包含 weixin 适配器依赖。
- 该 PRD 不改变用户使用 Hermes 模型的计费路径。用户对 HermesAgent 的调用仍由平台记录；Hermes 内部调用底层模型时仍建议使用独立服务账号或团队 Key，避免递归调用 Hermes 模型别名。

## Testing Decisions

- 测试应优先覆盖外部可见行为，不绑定前端组件内部状态或 Hermes 私有函数实现。
- 后端代理测试应覆盖：缺少 Hermes API key 返回服务不可用；Hermes 返回 400 时保留可读错误；正常创建技能返回成功；技能列表只返回当前用户可见内容。
- 技能隔离测试应构造两个不同平台用户，验证用户 A 创建的技能不会出现在用户 B 的“我的技能”列表。
- 工具列表测试应使用 Hermes `GET /v1/toolsets` 的代表性响应，验证平台代理和前端能显示启用、未启用、已配置、未配置和空工具列表状态。
- 微信扫码测试应使用可替代 Hermes weixin 适配器的测试桩，覆盖二维码生成、扫码中、已扫码、成功、过期、失败和断开连接状态。
- 前端测试应覆盖用户主流程：打开能力中心、创建技能、看到“我的技能”、查看工具集、发起微信扫码、二维码过期后重新生成。
- 多用户测试应覆盖同一浏览器切换账号和两个浏览器账号并发访问，确保技能、微信状态和本地缓存不会串号。
- 部署验证应覆盖 Hermes sidecar 启用和未启用两种情况。未启用时能力中心显示不可用说明；启用且版本正确时技能、工具和微信状态接口可用。
- 回归测试应确保 HermesAgent 聊天主流程不受能力中心失败影响：模型列表、发送消息、会话切换、`/new`、`/save`、`/retry` 仍可独立工作。

## Out of Scope

- 首期不实现完整 Hermes Desktop 的所有功能，例如本地文件管理、插件市场、全量配置编辑、自动更新、桌面通知和系统托盘。
- 首期不开放用户任意安装第三方工具或动态写入 MCP 服务器配置。
- 首期不实现团队共享技能市场。团队共享技能可以作为后续阶段，在权限、审核、版本和审计模型明确后再做。
- 首期不支持 Telegram、Discord、Slack、WhatsApp、企业微信、飞书、钉钉等其他消息平台。架构上保留扩展位，但产品验收只覆盖微信扫码。
- 首期不改变平台现有微信登录、微信公众号登录、微信支付配置。
- 首期不把 Hermes API Server 暴露给公网，也不允许普通用户直接持有 Hermes API Server 密钥。
- 首期不承诺微信个人号协议的可用性稳定性。需要在部署文档中说明该能力依赖 Hermes weixin 适配器和上游 iLink 服务状态。

## Further Notes

- 推荐交互结构：HermesAgent 页面保留聊天为主视图，能力中心使用紧凑的标签页或右侧抽屉，包含“我的技能”“工具”“消息平台”。不要做营销式页面，重点是可扫描、可操作、低干扰。
- “技能”和“工具”在界面上需要清晰区分：技能是给 Hermes 的可复用行为说明，工具是 Hermes 运行时可调用的能力集合。
- “微信扫码”需要与“微信登录”文案分开。前者连接 HermesAgent 消息平台，后者是白泽平台账号登录方式。
- 能力中心应优先中文体验，同时新增 UI 文案必须进入前端 i18n。
- 若 Hermes Desktop 后续新增稳定的能力管理 API，应优先复用其协议，减少 Web 版与 sidecar 的差异。

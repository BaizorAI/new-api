## 二、核心决策：混合架构（A + B 分层）

### 结论：不必二选一

经评估，**推荐 HermesAgent + 直接 LLM/API 调用分层混合**：

| 维度 | 路径 A：HermesAgent | 路径 B：直接 LLM | 推荐 |
|------|-------------------|-----------------|------|
| **剧本拆解** | Skill 定义 prompt + 输出格式，Agent 自动重试/修正 | Go 代码硬编码 prompt，需自行处理重试和格式校验 | A |
| **多步编排** | Agent 原生支持：拆解 → 规划 → 生成 → 校验 → 修复，全自动 | 需 Go 代码编排 pipeline，每步写状态机 | A |
| **工具调用** | 109+ 已有工具 + FFmpeg terminal + 图像/视频 API | 只能调 Baizor-API REST 端点，FFmpeg 需额外服务 | A |
| **角色一致性** | Agent 可跨镜头记忆上下文，自主决策是否需要修复 | 需 Go 代码显式传递上下文，逻辑复杂 | A |
| **异常处理** | Agent 可解释错误、自动调整参数重试 | 硬编码错误处理分支 | A |
| **实时交互** | Hermes WebSocket 流式输出，前端已有 Playground | 需新建 SSE/WS 接口 | A |
| **延迟** | Agent 启动 + LLM 多轮 ≈ 20-60s 额外开销 | 单次 LLM 调用 ≈ 3-10s | B |
| **简单任务** | 小题大做：单张图生成不需要 Agent | 直接调 API，毫秒级 | B |
| **可控性** | Agent 行为有随机性，prompt 调优周期长 | Go 代码完全确定性 | B |
| **部署复杂度** | 需 Python + Hermes 运行时 | 纯 Go，单二进制 | B |

### 分层架构

```
┌──────────────────────────────────────────────────────────────┐
│                       用户操作                                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  简单操作（Go 后端直接调用，≤1步）                              │
│  ├─ 单张图像生成         → /v1/images/generations             │
│  ├─ 单段视频生成         → /v1/videos/generations (Task异步)  │
│  ├─ 图像理解/描述       → /v1/chat/completions (VL模型)      │
│  └─ 剧本快速拆解        → /v1/chat/completions (structured)  │
│                                                              │

│  ├─ 剧本深度拆解+分镜规划   → script-analyzer Skill           │
```

## 三、Baizor-API 配置建议

### 3.1 模型注册（`common/model.go`）

**现状**：`huayu-drama-4` 仅在 `ImageGenerationModels` 中。

**建议**：huayu-drama-4 是全模态的统一品牌名。但底层实际是 4 个不同模型，建议在 Baizor-API 注册多个模型别名，映射到同一个 Channel（llama-proxy-huazhen），由 proxy 按路径自动路由。

```go
// common/model.go — 修改建议

var (
    ImageGenerationModels = []string{
        "dall-e-3",
        "dall-e-2",
        "gpt-image-1",
        "gpt-image-2",
        "huayu-drama-4",         // ← 保留，图像生成
        "prefix:imagen-",
        "flux-",
        "flux.1-",
    }

    // 新增：视频生成模型列表
    VideoGenerationModels = []string{
        "huayu-drama-4",         // 同名，由请求路径区分
        "huayu-drama-4-video",   // 或显式的视频模型别名
    }
)

// 新增
func IsVideoGenerationModel(modelName string) bool {
    modelName = strings.ToLower(modelName)
    for _, m := range VideoGenerationModels {
        if strings.Contains(modelName, m) {
            return true
        }
    }
    return false
}
```

### 3.2 Channel 配置（管理后台）

在 Baizor-API 管理后台添加 Channel：

```
通道名称：huazhen-cluster
类型：OpenAI (兼容)
Base URL：http://<anne-1-ip>:8080       ← llama-proxy-huazhen 对外端口
API Key：<LLAMA_API_KEY>
模型列表：huayu-drama-4,huazhen-v1,huazhen-vl,huazhen-fable-5
优先级：10
权重：100
分组：default
```

**关键**：由于 llama-proxy-huazhen 内部按路径路由，一个 Channel 即可覆盖 chat + image + video 所有请求。Baizor-API 只需按 model name 将请求路由到这个 Channel。

### 3.3 llama-proxy-huazhen 配置建议

#### 3.3.1 增加 `huayu-drama-4` 到 PROXY_MODELS

**当前**：
```yaml
- PROXY_MODELS=huazhen-v1,huazhen-vl,ideogram-4,sulphur-2
```

**建议**：
```yaml
- PROXY_MODELS=huayu-drama-4,huazhen-v1,huazhen-vl,ideogram-4,sulphur-2
```

并在 proxy.py 中将 `huayu-drama-4` 作为统一别名，所有 4 个能力的请求都接受 `model: huayu-drama-4`：
- `/v1/chat/completions` + model=huayu-drama-4 → 路由到 llama-huazhen
- `/v1/chat/completions` + model=huayu-drama-4 + 含图片 → 路由到 llama-vl-huazhen
- `/v1/images/generations` + model=huayu-drama-4 → 路由到 ideogram4-huazhen
- `/v1/videos/generations` + model=huayu-drama-4 → 路由到 sulphur2-huazhen

#### 3.3.2 新增异步视频 API 支持

**当前问题**：sulphur2 是同步阻塞（一个视频生成可能要 5-30 分钟），proxy 直接透传等待。对于 Baizor-API 的 Task 异步系统，建议两种方案：

**方案 A（推荐）：proxy 层新增异步封装**

在 llama-proxy-huazhen 中新增 OpenAI Videos API 异步模式支持：

```python
# proxy.py 新增路由
ALLOWED_PATHS.add("/v1/videos")           # POST 提交 + GET 查询
ALLOWED_PATHS.add("/v1/videos/{id}")      # GET 查询单个

# POST /v1/videos → 提交任务，立即返回 task_id
# GET /v1/videos/{id} → 查询任务状态/结果
```

请求格式（兼容 OpenAI Videos API）：
```json
POST /v1/videos
{
    "model": "huayu-drama-4",
    "input": [{"type": "text", "text": "A cinematic shot of..."}],
    "size": "1280x720",
    "duration": 5
}

→ 202 Accepted
{
    "id": "video_abc123",
    "status": "generating",
    "created_at": 1752600000
}
```

```json
GET /v1/videos/video_abc123

→ 200 OK (完成时)
{
    "id": "video_abc123",
    "status": "completed",
    "video": {"url": "https://..."},
    "created_at": 1752600000
}
```

**方案 B：Baizor-API 侧 Task 封装**

不改 proxy，在 Baizor-API 的 Task 系统中异步封装：
- `POST /v1/video/generations` → 创建 Task → 后台调 proxy 的同步 `/v1/videos/generations` → 完成后更新 Task
- 复用现有 `relay/channel/task/` 适配器模式

**推荐方案 A**，因为：
1. 与 Baizor-API 现有视频路由 (`/v1/videos`) 完全兼容
2. proxy 端可管理队列和并发，避免 ComfyUI 被压垮
3. Baizor-API 无需新建适配器

## 四、最高配建议（Q5）

### 4.1 视频生成参数 — 最高配

```yaml
# docker-compose.yml — sulphur2-huazhen 环境变量修改
- SULPHUR2_MAX_SIDE=1920          # 当前 1280 → 提升到 1080p
- SULPHUR2_NUM_FRAMES=193         # 当前 97 → 约 8 秒 @24fps
- SULPHUR2_FPS=24                 # 保持 24fps（电影标准）
- SULPHUR2_NUM_INFERENCE_STEPS=75 # 当前 50 → 更高质量
- SULPHUR2_GUIDANCE_SCALE=3.6     # 保持不变
```

| 参数 | 当前 | 最高配 | 影响 |
|------|------|--------|------|
| MAX_SIDE | 1280 | **1920** | 支持 1080p (1920×1080)，需更多显存 |
| NUM_FRAMES | 97 (~4s) | **193 (~8s)** | 单镜头 8 秒，影视场景够用 |
| FPS | 24 | **24** | 电影标准帧率 |
| NUM_INFERENCE_STEPS | 50 | **75** | 更细腻，生成时间 ×1.5 |
| 预计单视频耗时 | ~5-15 min | **~15-40 min** | GPU 4,5 两卡 |

**注意**：1920px + 193 帧可能在 2× RTX 4090 (48GB) 上 OOM。建议：
- 先试 `1280 + 193帧`（扩时长不扩分辨率）
- 再试 `1920 + 97帧`（扩分辨率不扩时长）
- 最后试 `1920 + 193帧`，如 OOM 则降回

### 4.3 Image-to-Video (i2v) 扩展方案

**目标**：支持先生成关键帧图像 → 再转视频，实现影视分镜 pipeline。

#### 方案：扩展 ComfyUI 工作流 + sulphur2 API

**Step 1：新增 i2v ComfyUI 工作流**

```
comfyui-workflows/
  ├─ ltx23_t2v_base_api.json      ← 现有 text-to-video
  └─ ltx23_i2v_base_api.json  🆕  ← 新增 image-to-video
```

LTX-Video 2.3 原生支持 image-to-video（`LTXVImgToVideoInplace` 节点），当前工作流中已有该节点（`NODE_BYPASS_I2V_22`），只是被 bypass 了。

**Step 2：修改 sulphur2/server.py**

```python
# sulphur2/server.py — 新增 i2v 端点

# 新增路由
ALLOWED_PATHS.add("/v1/videos/image-to-video")

# POST /v1/videos/image-to-video
# {
#     "model": "huayu-drama-4",
#     "image": "data:image/png;base64,...",  // 或 URL
#     "prompt": "the character walks forward...",
#     "size": "1280x720",
#     "num_frames": 97
# }
```

核心改动：
1. 取消 `NODE_BYPASS_I2V_22` 的 bypass（设为 false）
2. 将输入图片传入 ComfyUI 的 LoadImage 节点
3. 其余参数复用现有 t2v 逻辑

**Step 3：proxy 路由更新**

```yaml
# docker-compose.yml — llama-proxy-huazhen
- UPSTREAM_VIDEO_URL=http://sulphur2-huazhen:8080
# proxy.py 新增 ALLOWED_PATHS
# "/v1/videos/image-to-video"
```

**预计工作量**：2-3 天（ComfyUI 工作流已有 i2v 节点，只需开启 + API 封装）
